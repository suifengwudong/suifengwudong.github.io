# B 站视频编码

## 一、 启发与原因

在某一天后，我突然发现我无法使用 IDM（Internet Download Manager）下载 B 站的视频了，当时我以为是 IDM 的问题，于是我改用“bilibili 下载助手”这个浏览器扩展来下载视频，竟也相安无事。近日，我意识到功能的失效可能并不是 IDM 的问题（之后我会谈到 IDM 的下载机制），而是 B 站视频编码方式的改变导致的。于是，我开始研究 B 站视频编码的相关知识，并尝试找到解决方法。

## 二、 IDM 的下载机制

首先，对所有网页视频下载工具来说，**资源嗅探**是首要的技术。资源嗅探是指通过分析网页的 HTML、JavaScript 以及网络请求来识别视频资源的 URL。IDM 的嗅探主要依赖其浏览器扩展：当页面加载时，扩展会拦截所有网络请求，根据文件扩展名（如 `.mp4`、`.flv`、`.avi`）和MIME类型（事实上，这才是当下视频播放网页的主流）筛选出潜在的媒体资源，并在用户点击时弹出下载窗口。这些类型是预设的，因此，如果目标网站使用了新的编码格式或封装方式，导致资源的 URL 不再符合 IDM 浏览器扩展的嗅探规则，IDM 就无法识别这些资源并弹出“下载浮动条”。**但这并不意味着 IDM 无法下载这些资源**。简单来说，嗅探和下载是两个不同的过程：嗅探是识别资源的 URL，而下载则是根据 URL 将资源保存到本地。对 M4S 这类资源，IDM浏览器扩展默认不嗅探，但仍然是支持下载的。

> 如果想要直接了解如何操作，请跳转到后面的“让 IDM 继续下载 B 站视频的解决方案”部分。

## 三、M4S 文件格式与 DASH 流

### 3.1 流媒体协议

流媒体协议是指用于在互联网上传输音视频内容的协议，其核心是**自适应码率传输（ABR）**。目前全球流媒体领域存在两大核心协议：**HLS**（HTTP Live Streaming）和 **DASH**（Dynamic Adaptive Streaming over HTTP）。

#### 1. HLS 发展历程
HLS 是苹果公司主导设计的协议，现已成为全球流媒体领域的事实通用标准。
- **起源（2009年）**：由苹果随 iOS 3.0 推出，旨在解决传统 RTMP 协议在移动网络下的穿透性差和功耗高等问题。
- **演进（2010-2016年）**：从仅支持 TS 容器到 2016 年正式支持 **fMP4（分段 MP4）**，实现了与 DASH 的媒体格式互通。
- **标准化（2017年至今）**：发布 RFC 8216 标准，并推出**低延迟 HLS（LL-HLS）**，将延迟优化至 2 秒以内。

#### 2. DASH 发展历程
DASH（又称 MPEG-DASH）是由 MPEG 主导制定的**开放国际标准**，旨在打破私有协议的碎片化局面。
- **定义（2012年）**：ISO/IEC 23009-1:2012 正式发布，成为首个基于 HTTP 的自适应码率开放国际标准。
- **优势**：具有**编解码无关**和**厂商中立**的特性，支持 H.264/HEVC/AV1 等所有主流编码。
- **普及**：作为 YouTube、Netflix 及国内主流视频平台（如 B 站）的核心传输协议。

#### 3. 核心对比
| 维度 | HLS | DASH |
| :--- | :--- | :--- |
| 主导方 | 苹果公司 | MPEG 国际标准化组织 |
| 首次发布 | 2009 年 | 2012 年（正式标准） |
| 核心定位 | 从苹果生态延伸的事实标准 | 行业统一的开放国际标准 |

### 3.2 M4S（MPEG-4 Segment）结构

M4S 文件是使用 MPEG-DASH（基于 HTTP 的动态自适应流）技术通过互联网传输的音视频小片段。它基于 **ISO 基础媒体文件格式 (ISOBMFF)**，是将大型媒体文件（如 MP4）分段后生成的二进制数据块。

#### 1. M4S 与 MP4 的异同
M4S 和 MP4 都使用 **Box（或 Atom）结构**来存储元信息和媒体数据，但组织方式有所不同：
- **MP4 (Normal)**：通常是完整封存的文件。使用 `moov`（Movie Box）存储全局元数据（时间、播放顺序、轨道信息等），适合本地存储和渐进式下载。
```
ftyp  // 文件类型 Box，描述文件格式兼容性
moof  // 片段信息 Box，包含分段的元数据信息
  ├─ mfhd  // 片段头部，标识分段序号
  └─ traf  // 轨道片段，包含轨道信息
      ├─ tfhd  // 轨道片段头，包含轨道ID和基本信息
      └─ tfdt  // 轨道片段解码时间，用于同步
mdat  // 媒体数据 Box，包含音视频数据
```
- **M4S (Segmented)**：是分段存储的。它不包含 `moov`，而是使用 `moof`（Movie Fragment Box）来存储片段级的元数据。这种结构允许每个片段独立传输并播放，特别适合流媒体场景。
```
ftyp  // 文件类型标识
moov  // 全局媒体信息
  ├─ mvhd  // 电影头部，包含播放时长等信息
  ├─ trak  // 轨道数据
mdat  // 媒体数据
```

> 以上示例和分析来源于知乎文章，非常感谢原作者的详细分析。

#### 2. Box 内部结构
一个典型的 M4S 分体（fMP4）主要由以下 Box 组成：
- **ftyp (File Type Box)**：描述文件格式的兼容性。
- **styp (Segment Type Box)**：分段类型标识。
- **moof (Movie Fragment Box)**：片段信息，包含分段元数据。
    - **mfhd (Movie Fragment Header)**：标识分段序号。
    - **traf (Track Fragment)**：包含轨道片段的解码时间（`tfdt`）和头部信息（`tfhd`）。
- **mdat (Media Data Box)**：存储实际的经过编码的音视频帧。

#### 3. fMP4 与 M4S 的关系
- **fMP4 (Fragmented MP4)**：是指一种扩展的 MPEG-4 容器格式，允许将文件拆分为多个相互依赖或独立的片段。
- **M4S**：是 fMP4 格式在实际分段流（如 DASH 或 HLS）中的文件扩展名。简单来说，**fMP4 是容器标准，而 M4S 是该标准的物理表现形式**。

### 3.3 DASH 流的特点

根据维基百科及相关技术分析，MPEG-DASH 具有以下核心特点：

- **自适应比特率流**：DASH 将文件分割为多个片段，并提供不同码率的备选项。客户端播放器（如 `dash.js` 或 `Shaka Player`）会根据实时带宽自动在不同分辨率/比特率之间平滑切换。
- **编解码器中立**：DASH 标准本身不关心音视频的编解码格式。这意味着它可以支持任何编码格式（如 H.264、HEVC、AV1、VP9 等）进行分发。
- **MPD 清单文件**：使用 XML 格式的 `.mpd` 文件作为“作战地图”。它通过 `AdaptationSet` 和 `Representation` 描述了所有片段的 URL 路径、分辨率、码率等关键信息。
- **初始化与片段分离**：
    - **Initialization Segment (`init.m4s`)**：包含解码器初始化参数（如 SPS、PPS 等）。**播放任何片段前必须先下载它**。
    - **Media Segments (`chunk-*.m4s`)**：包含实际数据。通常第一个片段包含关键帧（I 帧），后续片段多为 P/B 帧。

### 3.4 B 站视频下载缓存方案的思考

B 站视频编码的演进历程可以总结为以下几个阶段：
1. 2015年，使用整段的FLV和MP4方案。这种方案随着视频时长增长，整段MP4的体积过大，导致加载极为缓慢；
2. 2016年，使用分段的FLV方案。这种方案简单且兼容性高，但在网络环境较差时，仍然会出现卡顿和缓冲问题；
3. 2020年，正式引入基于 DASH 的 M4S 分段方案。M4S 方案通过将视频分割成小片段，并根据网络状况动态调整质量，极大地提升了播放的流畅度和用户体验。

B 站目前在 Web 端和移动端全面采用了 DASH 技术，并对缓存文件进行了特定处理。

#### 1. 缓存文件目录结构
在 Windows 客户端或手机缓存目录中，通常会包含以下核心文件：
- **`video.m4s` 与 `audio.m4s`**：目前的 B 站策略是将完整的 DASH 流（多段 M4S）直接作为一个整体文件保存到本地，但依然保持了**音视频分离**的状态。
- **`playurl` (JSON)**：存储了各片段的 URL、清晰度映射、编解码参数（如 `avc1` 或 `hev1`）等。
- **`videoInfo`**：包含视频标题、UP 主信息及封面链接等元数据。
- **`danmaku.bin` / `dm1`**：B 站特有的二进制弹幕文件。

#### 2. 为什么是整体的 M4S 文件？
虽然在线播放时是分片的，但本地缓存将各片段合并为了一个单一的 `.m4s` 文件（包含 `init` 和所有 `media segments`）。
- **减少碎片**：防止离线状态下管理大量小文件带来的 IO 压力。
- **安全性与闭环**：通过分离存储和自定义文件头，增加了直接提取原始视频的门槛。

#### 3. 手动提取与合并
由于 B 站缓存的 `.m4s` 本质上仍是符合 ISOBMFF 规范合作的音视频流，我们可以利用 **FFmpeg** 将其还原为标准的 MP4 文件：

```bash
ffmpeg -i video.m4s -i audio.m4s -c copy output.mp4
```
生成的 `output.mp4` 即可被普通本地播放器识别。

## 四、 让 IDM 继续下载 B 站视频的解决方案

步骤如下：
1. 打开 IDM，点击“选项”（Options）选项卡。
2. 找到“自定义浏览器中的IDM下载浮动条”（Customize IDM Download panels in browsers）部分。
3. 在“文件类型”（File types）输入框中添加 `M4S`。
4. 点击“确定”保存设置后，刷新浏览器页面 IDM 即可自动弹出下载窗口。
